# ADR-0001: Parameterize Static Directory for Multi-Instance Deployment

## Status
Proposed

## Context

tex-server is a LaTeX PDF preview server with SyncTeX-based bidirectional search. The system consists of three layers:

1. **Compilation layer** (`compile_and_sync.sh`) — compiles LaTeX, outputs PDF and SyncTeX maps
2. **Server layer** (`pdf_server.py`) — serves PDF viewer, handles forward/reverse search via Socket.IO
3. **Frontend layer** (JS in `static/tex-server/`) — PDF.js viewer with sync capabilities

Currently the static directory is hardcoded in two places:

| File | Line | Hardcoded Value | Role |
|------|------|----------------|------|
| `compile_and_sync.sh` | 42 | `OUTPUT_DIR="$SCRIPT_DIR/static"` | Compilation output destination |
| `pdf_server.py` | 51 | `os.path.join(os.path.dirname(...), 'static')` | Default when no `static_dir` argument passed |

This means only one workspace can exist per installation. When multiple agents need independent compilation targets and PDF viewers simultaneously, the single `static/` directory becomes a bottleneck — agents would overwrite each other's `main.pdf`, `main.synctex.gz`, and JSON map files.

### What lives in the static directory

Per-workspace (must be isolated):
- `main.pdf` — compiled PDF
- `main.synctex.gz` — SyncTeX data
- `forward_map.json`, `reverse_map.json`, `file_map.json` — lookup maps generated by `synctex_tool.py`

Shared infrastructure (identical across all instances):
- `pdfjs/` — PDF.js viewer and `viewer_patched.html`
- `tex-server/` — frontend JS (`config.js`, `sync_socket_io.js`, `double_click_page_position.js`)

### What is already parameterized

`pdf_server.py:49` `create_app(static_dir=None)` already accepts a custom directory. The reverse search handler at lines 105-109 reads JSON maps from `static_dir` via closure — no hardcoding there. The `synctex_tool.py` CLI accepts `--json-dir` and `--synctex` as explicit paths — fully parameterized. Frontend JS uses `window.location.origin` for the server URL — no hardcoded host/port.

### What is NOT parameterized

1. `pdf_server.py` `__main__` block (line 160) calls `create_app()` with no arguments — no CLI flag to pass a custom static dir
2. `compile_and_sync.sh` line 42 hardcodes `OUTPUT_DIR="$SCRIPT_DIR/static"` — no parameter to override
3. `compile_and_sync.sh` reads port from a single `config.ini` — cannot target a different server instance

## Research

### Multi-instance vs multi-tenant

Two standard patterns for serving isolated workspaces:

**Multi-tenant (single process, routing by workspace ID):**
Requires workspace-aware routing, Socket.IO room/namespace isolation, per-workspace state management. Every layer (routes, Socket.IO protocol, frontend JS, state dict) needs modification. Complexity is proportional to number of layers.

**Multi-instance (one process per workspace):**
Each process serves one workspace. Isolation is provided by the OS (separate process, separate port). No protocol changes needed. Complexity is constant regardless of workspace count.

Since workspaces have zero interaction — no shared state, no cross-workspace queries, no common dashboard — multi-tenant adds complexity with no benefit. Multi-instance leverages OS-level isolation for free.

### Resource cost

Flask + Socket.IO process: ~20-30MB RSS per instance. For 10 agents: ~200-300MB. Negligible on any modern machine.

## Decision

**Parameterize the static directory as a CLI argument**, enabling multi-instance deployment where each agent runs its own server process with its own static directory and port.

Changes required:

### 1. `pdf_server.py` — add `--static-dir` CLI argument

`__main__` block adds `--static-dir` to `argparse`, passes value to `create_app(static_dir=args.static_dir)`. When omitted, falls back to current default (`<script_dir>/static`).

Port already comes from `config.ini` or could be added as `--port` override for convenience (avoids needing a separate `config.ini` per instance).

### 2. `compile_and_sync.sh` — add output directory and port parameters

New parameters (backward-compatible — all optional with current defaults):

- `$5` (or a named flag) = output directory for PDF, synctex, and JSON maps. Default: `$SCRIPT_DIR/static`
- `$6` (or a named flag) = server port to curl. Default: read from `config.ini`

Lines affected:
- Line 42: `OUTPUT_DIR` reads from parameter instead of hardcoded path
- Line 33-38: port can be overridden by parameter
- Lines 119-120, 148-150, 159-160, 166-170, 186-189: all already use `$OUTPUT_DIR` — no additional changes

### 3. Workspace directory setup

Each workspace directory needs `pdfjs/` and `tex-server/` subdirectories to serve the viewer and frontend JS. These are shared and identical across instances — use symlinks:

```bash
mkdir -p /path/to/myworkspace
ln -s /path/to/tex-server/static/pdfjs /path/to/myworkspace/pdfjs
ln -s /path/to/tex-server/static/tex-server /path/to/myworkspace/tex-server
```

This can be a small helper script, or left as a manual convention.

### No changes required

| Component | Why |
|-----------|-----|
| `synctex_tool.py` | Already accepts `--synctex` and `--json-dir` as explicit paths |
| `socketapp.py` | Global state is per-process — multi-instance isolates it automatically |
| `send_socket_message_to_pdfjs.py` | Stateless; reads from request params, emits to socketio |
| `static/tex-server/config.js` | Uses `window.location.origin` — adapts to any host:port |
| `static/tex-server/sync_socket_io.js` | Receives coordinates from Socket.IO, no hardcoded paths |
| `static/tex-server/double_click_page_position.js` | Emits click coordinates back to server, no hardcoded paths |
| `pdf_server.py` reverse search (lines 105-109) | Already uses `static_dir` from `create_app` closure |
| `pdf_server.py` index route (line 131) | `/static/` is Flask URL prefix mapped to `static_dir` — works with any directory |

## Alternatives Considered

### A. Single-process multi-workspace with URL routing

Route by workspace ID: `/<workspace_id>/`, Socket.IO rooms per workspace, per-workspace state dict.

Rejected because:
- Requires changes in **every layer**: routing, Socket.IO protocol, frontend JS (parse workspace from URL, filter messages), state management
- Workspaces have zero interaction — multi-tenant adds ~50-80 lines of routing/room logic for no functional benefit
- Debugging harder: all workspace logs in one process, one crash affects all workspaces
- Python GIL means concurrent compilation requests across workspaces contend anyway

### B. Subdirectory convention within single static dir

`static/workspaces/agent-1/`, `static/workspaces/agent-2/`, served by one process.

Rejected because:
- Still requires URL routing changes (`?file=static/workspaces/agent-1/main.pdf`)
- Frontend JS needs to know its workspace path
- Reverse search path construction becomes workspace-aware
- Gains nothing over multi-instance since workspaces don't interact

## Consequences

### Positive
- Multi-agent support with minimal code change (~10 lines across 2 files)
- Full process isolation — one agent's crash/hang does not affect others
- Zero protocol changes — Socket.IO, frontend JS, reverse search all unchanged
- Backward compatible — omitting `--static-dir` preserves current behavior

### Negative
- Each instance occupies a port — need a port allocation convention (e.g., base + N)
- Each workspace directory needs symlinks to `pdfjs/` and `tex-server/` — small setup cost
- No single dashboard listing all active workspaces (can be added later as a separate lightweight service if needed)

### Risks
- Flask follows symlinks by default, but if the deployment environment restricts symlink traversal, the shared `pdfjs/` and `tex-server/` won't load. Mitigation: test symlink serving on target environment; fallback is copying the directories.

## Affected Scope

| File | Change |
|------|--------|
| `pdf_server.py` | `__main__`: add `--static-dir` and optionally `--port` to argparse |
| `compile_and_sync.sh` | Add output-dir and port parameters, update `OUTPUT_DIR` assignment |
| (new, optional) `init_workspace.sh` | Helper to create workspace dir with symlinks |
